System Architecture

Collaborative Document Engine

A Spring Boot based collaborative editing platform that combines durable document persistence in PostgreSQL with low-latency multi-instance collaboration over STOMP/WebSocket, Redis Pub/Sub fanout, and Kafka-backed accepted-operation streaming for replay and downstream processing.

Primary users Product teams, interviewers, and engineers evaluating real-time collaborative editing design.
Business value Demonstrates a defensible split between hot-path collaboration, durable storage, and async event flows.
Design target Horizontally scalable backend with ordered operations, ACL enforcement, and recoverable edit history.
Current stack React/Vite frontend, Spring Boot backend, PostgreSQL, Redis, Kafka, Flyway, Docker Compose.

1. Business Context

What the system exists to do and how success is measured.

Purpose Enable multiple users to read, edit, and share structured documents in real time while the server remains the ordering authority for accepted changes.
Target users Owners creating documents, collaborators with role-based access, and teams needing a fast but auditable collaboration backbone.
Main objectives Preserve document integrity, keep collaboration responsive across instances, and provide a system that is simple to explain in interviews and practical to evolve.
Expected outcomes Low-latency accepted-operation delivery, durable edit history, ACL enforcement across REST and messaging, and clear infrastructure responsibility boundaries.
Data ownership in PostgreSQL Server-authoritative OT / rebase Low-latency collaboration Durable event stream for replay

2. Data Flow Diagram

Hot path for create, join, edit, fanout, and durable event publishing.

The edit hot path stays server-controlled: a client submits an operation with operationId and baseVersion, the backend validates access, transforms as needed, persists the accepted result, then publishes to Redis for low-latency fanout and Kafka for durability.

3. Processing Pipeline

The ordered stages of a collaboration request.

1. Ingress REST calls handle document CRUD and sharing. STOMP endpoints handle join, leave, presence, and edit submission.
2. Identity & ACL Spring Security authenticates HTTP and WebSocket traffic, then document-specific authorization checks gate read or write access.
3. Validation Operation shape, session state, and collaborator semantics are validated before the write path proceeds.
4. Ordering & Rebase The backend locks the document row, checks idempotency, loads intervening operations, and rebases or converts to NO_OP when needed.
5. Persistence The materialized document snapshot and immutable operation record are saved in PostgreSQL under one transactional flow.
6. Distribution Local subscribers receive immediate topic broadcasts, Redis propagates to other instances, and Kafka receives an after-commit accepted-operation event.

4. System Architecture

Layered view from browser to infrastructure.

Client and editor The frontend uses React, Vite, Tiptap, SockJS, and STOMP to combine metadata management with live editing.
Backend responsibility Spring Boot owns request handling, authorization, operation ordering, document projection updates, and downstream publication.
State boundaries PostgreSQL is the source of truth, Redis carries transient coordination and fanout, and Kafka holds durable accepted-operation events.
Scalability posture Any backend instance can accept a client connection, while Redis and Kafka let collaboration and event processing extend beyond one node.

5. Features

Functional and non-functional requirements represented by the current codebase.

Functional features
  • Document CRUD and paginated listing for accessible resources.
  • Ownership transfer and collaborator permission management.
  • Real-time join, leave, presence, and accepted-operation messaging.
  • Operation log plus materialized current document state.
  • Search and filtering support for document discovery.
Authorization model
  • Owner, collaborator, and visibility-based access semantics.
  • Read authorization enforced for subscriptions and session joins.
  • Write authorization enforced before document mutations and operation submission.
  • Per-user access-revocation topics for real-time removal flows.
Reliability characteristics
  • Idempotency checks on operationId avoid duplicate accepted operations.
  • Pessimistic locking serializes version assignment on the document aggregate.
  • Kafka publication occurs after commit to avoid durable events for rolled-back writes.
  • Redis propagation filters same-instance echoes to prevent duplicate broadcasts.
Operational qualities
  • Flyway-controlled schema evolution with ddl-auto=validate.
  • Actuator health and Prometheus endpoints for runtime visibility.
  • Automated test coverage across controllers, services, Redis, Kafka, and end-to-end collaboration paths.
  • Docker Compose local environment spanning backend, frontend, PostgreSQL, Redis, and Kafka.

6. Deployment

How the system is run locally today and how it is intended to scale.

Local development
  • docker compose up -d starts PostgreSQL, Redis, Kafka, backend, and frontend.
  • The backend container exposes port 8080; the frontend serves on 3000.
  • Java 17 and Maven wrapper support local backend execution outside containers as well.
Backend packaging
  • A two-stage Dockerfile builds the Spring Boot jar with Maven, then runs it on Eclipse Temurin JRE 17.
  • Environment variables inject database, Redis, Kafka, and JWT settings.
  • Flyway migrations run at startup and validate the schema shape.
Runtime dependencies
  • PostgreSQL stores durable document and operation state.
  • Redis supports cross-instance fanout, presence, and session propagation.
  • Kafka provides a durable document-operations stream for notifications, replay, and analytics consumers.
Scale-out direction
  • Multiple backend instances can serve clients because fanout is not tied to one node.
  • Redis keeps collaboration fast; Kafka keeps downstream processing durable.
  • The architecture is ready for future production concerns such as stricter JWT enforcement and more advanced async consumers.

Current identity handling mixes an X-User-Id oriented development flow with JWT-capable security components. In the architecture story, this is best described as a pragmatic MVP authentication layer with a clearer JWT-first production path.
